## 简介

技能是一组指令，打包成一个简单的文件夹，教会 Claude 如何处理特定任务或工作流。技能是针对您的特定需求定制 Claude 的最强大方式之一。无需在每次对话中重新解释您的偏好、流程和领域专业知识，技能让您只需教会 Claude 一次，就能在每次使用时受益。

当您有可重复的工作流时，技能非常强大：根据规范生成前端设计，使用一致的方法进行研究，创建遵循团队风格指南的文档，或编排多步骤流程。它们与 Claude 的内置功能（如代码执行和文档创建）配合良好。对于构建 MCP 集成的用户，技能增加了另一个强大的层次，帮助将原始工具访问转变为可靠、优化的工作流。

本指南涵盖了构建有效技能所需的所有知识，从规划和结构到测试和分发。无论您是为自己、团队还是社区构建技能，您都会发现实用的模式和真实世界的示例。

您将学到：

- 技能结构的技术要求和最佳实践
- 独立技能和 MCP 增强工作流的模式
- 我们在不同用例中看到效果良好的模式
- 如何测试、迭代和分发您的技能

本书适用于：

- 希望 Claude 一致遵循特定工作流的开发人员
- 希望 Claude 遵循特定工作流的高级用户
- 希望标准化 Claude 在其组织中工作方式的团队

### 本指南的两种路径

构建独立技能？重点关注基础、规划与设计以及第1-2章。增强 MCP 集成？"技能 + MCP" 部分和第3章适合您。两条路径共享相同的技术要求，但您可以选择与您的用例相关的内容。

您将从本指南中获得：结束时，您将能够一次性构建一个功能完整的技能。预计使用 skill-creator 构建和测试第一个可工作的技能需要 15-30 分钟。

让我们开始吧。

# 第1章：基础

## 什么是技能？

技能是一个包含以下内容的文件夹：

- `SKILL.md`（必需）：带有 YAML frontmatter 的 Markdown 指令
- `scripts/`（可选）：可执行代码，如 Python 或 Bash
- `references/`（可选）：按需加载的文档
- `assets/`（可选）：输出中使用的模板、字体、图标

## 核心设计原则

### 渐进式披露

技能使用三级系统：

- 第一级（YAML frontmatter）：始终加载在 Claude 的系统提示中。提供足够的信息让 Claude 知道何时应使用每个技能，而无需将所有内容加载到上下文中。
- 第二级（`SKILL.md` 正文）：当 Claude 认为技能与当前任务相关时加载。包含完整的指令和指导。
- 第三级（链接文件）：技能目录中捆绑的附加文件，Claude 可以选择仅在需要时导航和发现。

这种渐进式披露最小化了令牌使用，同时保持了专业知识。

### 可组合性

Claude 可以同时加载多个技能。您的技能应与其他技能良好协作，而不是假设它是唯一可用的功能。

### 可移植性

技能在 Claude.ai、Claude Code 和 API 上工作方式相同。创建一次技能，它可以在所有平台上工作而无需修改，前提是环境支持技能所需的任何依赖项。

## 针对 MCP 构建者：技能 + 连接器

在没有 MCP 的情况下构建独立技能？跳到规划与设计；您可以稍后返回此处。

如果您已经有了一个可工作的 MCP 服务器，那么您已经完成了困难的部分。技能是顶层的知识层，捕获您已经知道的工作流和最佳实践，以便 Claude 可以一致地应用它们。

### 厨房类比

MCP 提供专业厨房：工具、食材和设备的访问权限。

技能提供食谱：如何创建有价值的东西的分步说明。

它们共同使用户能够完成复杂的任务，而无需自己弄清楚每个步骤。

### 它们如何协同工作

| MCP (Connectivity) | Skills (Knowledge) |
| --- | --- |
| 将 Claude 连接到您的服务，如 Notion、Asana 或 Linear | 教会 Claude 如何有效使用您的服务 |
| 提供实时数据访问和工具调用 | 捕获工作流和最佳实践 |
| Claude 可以做什么 | Claude 应该怎么做 |

### 为什么这对您的 MCP 用户很重要

没有技能：

- 用户连接您的 MCP 但不知道下一步该做什么
- 支持工单询问"如何使用您的集成做 X"
- 每次对话都从头开始
- 结果不一致，因为用户每次提示的方式不同
- 当真正的问题是工作流指导时，用户会责怪您的连接器

有技能：

- 预构建的工作流在需要时自动激活
- 一致、可靠的工具使用
- 最佳实践嵌入到每次交互中
- 更低的学习曲线来使用您的集成

# 第二章：规划与设计

## 从用例开始

在编写任何代码之前，确定你的技能应该实现的 2-3 个具体用例。

良好的用例定义：

```text
用例：项目冲刺规划
触发：用户说"帮我规划这次冲刺"或"创建冲刺任务"
步骤：
1. 从 Linear 获取当前项目状态（通过 MCP）
2. 分析团队速度和容量
3. 建议任务优先级
4. 在 Linear 中创建任务，添加适当的标签和估算
结果：完全规划的冲刺，任务已创建
```

问自己：

- 用户想要完成什么？
- 这需要哪些多步骤工作流？
- 需要哪些工具，内置的还是 MCP？
- 应该嵌入哪些领域知识或最佳实践？

## 常见技能用例类别

在 Anthropic，我们观察到三种常见的用例。

### 类别 1：文档和资产创建

用于创建一致的高质量输出，包括文档、演示文稿、应用、设计、代码和类似工件。

真实示例：frontend-design 技能，另请参见 docx、pptx、xlsx 和 ppt 的技能。

> 创建具有高质量设计特色的、生产级的前端界面。在构建 Web 组件、页面、工件、海报或应用程序时使用。

关键技术：

- 嵌入式风格指南和品牌标准
- 用于一致输出的模板结构
- 最终确定前的质量检查清单
- 不需要外部工具；使用 Claude 的内置功能

### 类别 2：工作流自动化

用于从一致方法中受益的多步骤流程，包括跨多个 MCP 服务器的协调。

真实示例：skill-creator 技能。

> 创建新技能的交互式指南。引导用户完成用例定义、frontmatter 生成、指令编写和验证。

关键技术：

- 带有验证门的工作流步骤
- 常见结构的模板
- 内置的审查和改进建议
- 迭代改进循环

### 类别 3：MCP 增强

用于增强 MCP 服务器提供的工具访问的工作流指导。

真实示例：来自 Sentry 的 sentry-code-review 技能。

> 通过其 MCP 服务器使用 Sentry 的错误监控数据自动分析和修复 GitHub 拉取请求中检测到的错误。

关键技术：

- 按顺序协调多个 MCP 调用
- 嵌入领域专业知识
- 提供用户通常需要指定的上下文
- 常见 MCP 问题的错误处理

## 定义成功标准

你如何知道你的技能正在工作？

这些是目标指标：粗略的基准而非精确的阈值。力求严谨，但要接受会有基于感觉的评估因素。我们正在积极开发更强大的测量指导和工具。

定量指标：

- 技能在 90% 的相关查询中触发
  - 如何衡量：运行 10-20 个应该触发你的技能的测试查询。跟踪它自动加载的次数与需要显式调用的次数。
- 在 X 次工具调用中完成工作流
  - 如何衡量：比较启用和禁用技能时的相同任务。计算工具调用和消耗的总令牌数。
- 每个工作流 0 次 API 调用失败
  - 如何衡量：在测试运行期间监控 MCP 服务器日志。跟踪重试率和错误代码。

定性指标：

- 用户不需要提示 Claude 下一步做什么
  - 如何评估：在测试期间，记录你需要重定向或澄清的频率。向 beta 用户征求反馈。
- 工作流无需用户纠正即可完成
  - 如何评估：对同一请求运行 3-5 次。比较输出的结构一致性和质量。
- 跨会话的一致结果
  - 如何评估：新用户能否在首次尝试时在最少指导下完成任务？

## 技术要求

### 文件结构

```text
your-skill-name/
├── SKILL.md              # 必需 - 主要技能文件
├── scripts/              # 可选 - 可执行代码
│   ├── process_data.py   # 示例
│   └── validate.sh       # 示例
├── references/           # 可选 - 文档
│   ├── api-guide.md      # 示例
│   └── examples/         # 示例
└── assets/               # 可选 - 模板等
    └── report-template.md # 示例
```

### 关键规则

`SKILL.md` 命名：

- 必须是 `SKILL.md`（区分大小写）
- 不接受变体，如 `SKILL.MD` 或 `skill.md`

技能文件夹命名：

- 使用 kebab-case：`notion-project-setup`
- 无空格：`Notion Project Setup` 无效
- 无下划线：`notion_project_setup` 无效
- 无大写：`NotionProjectSetup` 无效

无 `README.md`：

- 不要在技能文件夹内包含 `README.md`。
- 所有文档都放在 `SKILL.md` 或 `references/` 中。
- 注意：通过 GitHub 分发时，你仍需要为人类用户提供仓库级别的 README。请参阅分发和共享。

### YAML Frontmatter：最重要的部分

YAML frontmatter 是 Claude 决定是否加载你的技能的方式。确保正确设置。

最小必需格式：

```yaml
---
name: your-skill-name
description: 它的功能。当用户要求[特定短语]时使用。
---
```

这就是你开始所需的一切。

字段要求：

`name`（必需）：

- 仅使用 kebab-case
- 无空格或大写
- 应与文件夹名称匹配

`description`（必需）：

- 必须包括：
  - 技能的功能
  - 何时使用（触发条件）
- 少于 1024 个字符
- 无 XML 标记（`<` 或 `>`）
- 包括用户可能说的具体任务
- 如相关，提及文件类型

`license`（可选）：

- 如果使技能开源，请使用
- 常见值：MIT、Apache-2.0

`compatibility`（可选）：

- 1-500 个字符
- 指示环境要求，如预期产品、必需的系统软件包或网络访问需求

`metadata`（可选）：

- 任何自定义键值对
- 建议：author、version、mcp-server

示例：

```yaml
metadata:
  author: ProjectHub
  version: 1.0.0
  mcp-server: projecthub
```

### 安全限制

Frontmatter 中禁止：

- XML 尖括号（`<` 或 `>`）
- 名称中包含 "claude" 或 "anthropic" 的技能；这些是保留的

原因：Frontmatter 出现在 Claude 的系统提示中。恶意内容可能注入指令。

## 编写有效的技能

### 描述字段

根据 Anthropic 的工程博客，此元数据为 Claude 提供了足够的信息，使其知道何时应使用每个技能，而无需将所有内容都加载到上下文中。这是渐进式披露的第一级。

结构：

```
[功能] + [何时使用] + [关键能力]
```

良好描述的示例：

```yaml
# 良好 - 具体且可操作
description: 分析 Figma 设计文件并生成开发者交接文档。当用户上传 .fig 文件，要求"设计规范"、"组件文档"或"设计到代码交接"时使用。

# 良好 - 包含触发短语
description: 管理 Linear 项目工作流，包括冲刺规划、任务创建和状态跟踪。当用户提到"冲刺"、"Linear 任务"、"项目规划"或要求"创建工单"时使用。

# 良好 - 清晰的价值主张
description: PayFlow 的端到端客户入职工作流。处理账户创建、支付设置和订阅管理。当用户说"入职新客户"、"设置订阅"或"创建 PayFlow 账户"时使用。
```

不良描述的示例：

```yaml
# 太模糊
description: 帮助处理项目。

# 缺少触发器
description: 创建复杂的多页文档系统。

# 太技术化，无用户触发器
description: 实现具有层次关系的项目实体模型。
```

在 frontmatter 之后，用 Markdown 编写实际指令。

推荐结构：

为你的技能调整此模板。将括号部分替换为你的具体内容。

````markdown
---
name: your-skill
description: [--.]
---

### 步骤 1: [第一个主要步骤]

对发生的事情进行清晰解释。

示例：

```bash
python scripts/fetch_data.py --project-id PROJECT_ID
```

预期输出：[描述成功时的样子]

(根据需要添加更多步骤。)

## 示例

用户说："设置一个新的营销活动"

操作：

1. 通过 MCP 获取现有活动
2. 使用提供的参数创建新活动

结果：活动已创建，包含确认链接

(根据需要添加更多示例。)

## 故障排除

错误：[常见错误信息]

原因：[为什么会发生]

解决方案：[如何修复]

(根据需要添加更多错误案例。)

### 指令的最佳实践

要具体且可操作。

好的：

```text
运行 `python scripts/validate.py --input {filename}` 检查数据格式。

如果验证失败，常见问题包括：
- 缺少必填字段（添加到 CSV 中）
- 无效的日期格式（使用 YYYY-MM-DD）
```

不好的：

```text
在继续前验证数据。
```

包含错误处理：

```markdown
## 常见问题

### MCP 连接失败

如果您看到"连接被拒绝"：

1. 验证 MCP 服务器正在运行：检查设置 > 扩展程序。
2. 确认 API 密钥有效。
3. 尝试重新连接：设置 > 扩展程序 > [您的服务] > 重新连接。
```

清晰地引用捆绑资源：

```markdown
在编写查询前，查阅 `references/api-patterns.md` 了解：

- 速率限制指导
- 分页模式
- 错误代码和处理
```

使用渐进式披露：

保持 `SKILL.md` 专注于核心指令。将详细文档移至 `references/` 并链接到它。有关三级系统的工作原理，请参阅核心设计原则。

# 第 3 章：测试和迭代

技能可以根据您的需求以不同程度的严格性进行测试：

- Claude.ai 中的手动测试：直接运行查询并观察行为。快速迭代，无需设置。
- Claude Code 中的脚本测试：自动化测试用例，以便在变更中进行可重复的验证。
- 通过技能 API 进行编程测试：构建评估套件，针对定义的测试集进行系统性运行。

选择符合您的质量要求和技能可见性的方法。由小团队内部使用的技能与部署到数千个企业用户的技能具有不同的测试需求。

## 专业提示：在扩展前迭代单个任务

我们发现，最有效的技能创建者会迭代单个具有挑战性的任务，直到 Claude 成功，然后将获胜的方法提取到技能中。这利用了 Claude 的上下文学习，并提供比广泛测试更快的信号。一旦您有了工作基础，扩展到多个测试用例以获得覆盖范围。

## 推荐的测试方法

基于早期经验，有效的技能测试通常涵盖三个领域。

### 1. 触发测试

目标：确保您的技能在正确的时间加载。

测试用例：

- 在明显的任务上触发
- 在改写的请求上触发
- 在不相关的主题上不触发

示例测试套件：

应该触发：

```text
Help me set up a new ProjectHub workspace
I need to create a project in ProjectHub
Initialize a ProjectHub project for Q4 planning
```

不应该触发：

```text
What's the weather in San Francisco?
Help me write Python code
Create a spreadsheet
```

除非 ProjectHub 技能处理表格，否则不应触发电子表格示例。

### 2. 功能测试

目标：验证技能产生正确的输出。

测试用例：

- 生成有效输出
- API 调用成功
- 错误处理有效
- 涵盖边缘情况

示例：

```text
测试：创建包含 5 个任务的项目

给定：项目名称"Q4 规划"，5 个任务描述
当：技能执行工作流
然后：
  - 在 ProjectHub 中创建项目
  - 创建 5 个具有正确属性的任务
  - 所有任务链接到项目
  - 无 API 错误
```

### 3. 性能比较

目标：证明技能相比基线改进了结果。

使用"定义成功标准"中的指标。以下是可能的比较方式。

基线比较：

没有技能：

- 用户每次提供指令
- 15 条来回消息
- 3 次失败的 API 调用需要重试
- 消耗 12,000 个 token

有技能：

- 自动工作流执行
- 仅 2 个澄清问题
- 0 次失败的 API 调用
- 消耗 6,000 个 token

## 使用技能创建者技能

技能创建者技能可通过 Claude.ai 的插件目录或 Claude Code 的下载获得，可帮助您构建和迭代技能。如果您有 MCP 服务器并了解您的 2-3 个主要工作流，您可以在一次会话中构建和测试一个功能技能，通常需要 15-30 分钟。

创建技能：

- 从自然语言描述生成技能
- 生成带有 frontmatter 的正确格式 `SKILL.md`
- 建议触发短语和结构

审查技能：

- 标记常见问题，如模糊描述、缺少触发器和结构问题
- 识别潜在的过度触发或触发不足风险
- 根据技能的陈述目的建议测试用例

迭代改进：

- 使用您的技能并遇到边缘情况或故障后，将这些示例带回技能创建者。
- 示例："使用此聊天中确定的问题和解决方案来改进技能如何处理[特定边缘情况]。"

使用方法：

```text
Use the skill-creator skill to help me build a skill for [your use case].
```

注意：skill-creator 帮助您设计和完善技能，但不执行自动化测试套件或产生定量评估结果。

## 基于反馈的迭代

技能是活的文档。计划基于以下内容进行迭代：

触发不足信号：

- 技能在应该加载时没有加载
- 用户手动启用它
- 关于何时使用它的支持问题

解决方案：在描述中添加更多细节和细微差别。这可能包括关键词，特别是对于技术术语。

过度触发信号：

- 技能为不相关的查询加载
- 用户禁用它
- 对目的感到困惑

解决方案：添加负面触发器并更加具体。

执行问题：

- 结果不一致
- API 调用失败
- 需要用户纠正

解决方案：改进指令并添加错误处理。

# 第5章：模式与故障排除

这些模式来自早期采用者和内部团队创建的技能。它们代表了我们看到行之有效的常见方法，而非规定性的模板。

## 选择您的方法：问题优先与工具优先

想象一下 Home Depot 的场景。您可能带着问题走进去，"我需要修理一个厨房橱柜"，然后员工会为您指向正确的工具。或者您可能挑选一把新钻头，询问如何将其用于您特定的工作。

技能的工作方式相同：

- 问题优先："我需要设置一个项目工作区" -> 您的技能以正确的顺序编排适当的 MCP 调用。用户描述结果；技能处理工具。
- 工具优先："我已连接 Notion MCP" -> 您的技能向 Claude 教授最佳工作流程和最佳实践。用户拥有访问权限；技能提供专业知识。

大多数技能倾向于一个方向。了解哪种框架适合您的用例可以帮助您选择以下正确的模式。

## 模式1：顺序工作流编排

使用场景：您的用户需要按特定顺序执行多步骤流程。

示例结构：

```markdown
# 工作流：新客户入职

## 步骤1：创建账户

调用 MCP 工具：`create_customer`
参数：姓名、电子邮件、公司

## 步骤2：设置支付

调用 MCP 工具：`setup_payment_method`
等待：支付方式验证

## 步骤3：创建订阅

调用 MCP 工具：`create_subscription`
参数：plan_id、customer_id（来自步骤1）

## 步骤4：发送欢迎邮件

调用 MCP 工具：`send_email`
模板：welcome_email_template
```

关键技巧：

- 明确的步骤排序
- 步骤之间的依赖关系
- 每个阶段的验证
- 失败时的回滚说明

## 模式2：多 MCP 协调

使用场景：工作流程跨越多个服务。

示例：设计到开发的交接。

```markdown
## 阶段1：设计导出（Figma MCP）

1. 从 Figma 导出设计资源。
2. 生成设计规范。
3. 创建资源清单。

## 阶段2：资源存储（Drive MCP）

1. 在 Drive 中创建项目文件夹。
2. 上传所有资源。
3. 生成可共享链接。

## 阶段3：任务创建（Linear MCP）

1. 创建开发任务。
2. 将资源链接附加到任务。
3. 分配给工程团队。

## 阶段4：通知（Slack MCP）

1. 将交接摘要发布到 #engineering 频道。
2. 包含资源链接和任务引用。
```

关键技巧：

- 清晰的阶段分离
- MCP 之间的数据传递
- 移至下一阶段前的验证
- 集中式错误处理

## 模式3：迭代优化

使用场景：输出质量随迭代而提高。

示例：报告生成。

```markdown
# 迭代式报告创建

## 初稿

1. 通过 MCP 获取数据。
2. 生成初稿报告。
3. 保存到临时文件。

## 质量检查

1. 运行验证脚本：`scripts/check_report.py`。
2. 识别问题：

   - 缺失部分
   - 格式不一致
   - 数据验证错误

## 优化循环

1. 解决每个已识别的问题。
2. 重新生成受影响的章节。
3. 重新验证。
4. 重复直到达到质量阈值。

## 最终确定

1. 应用最终格式。
2. 生成摘要。
3. 保存最终版本。
```

关键技巧：

- 明确的质量标准
- 迭代改进
- 验证脚本
- 知道何时停止迭代

## 模式4：上下文感知的工具选择

使用场景：相同的结果，根据上下文使用不同的工具。

示例：文件存储。

```

关键技巧：

- 清晰的决策标准
- 备选方案
- 选择透明度

## 故障排除

### 技能无法上传

错误："Could not find SKILL.md in uploaded folder"（在上传的文件夹中找不到SKILL.md）

原因：文件名不完全为`SKILL.md`。

解决方法：

- 重命名为`SKILL.md`（区分大小写）。
- 使用`ls -la`验证；它应显示`SKILL.md`。

错误："Invalid frontmatter"（无效的frontmatter）

原因：YAML格式问题。

常见错误：

```yaml


# 错误 - 缺少分隔符
name: my-skill
description: Does things



# 错误 - 未闭合的引号
name: my-skill
description: "Does things



# 正确
---
name: my-skill
description: Does things
---
```

错误："Invalid skill name"（无效的技能名称）

原因：名称包含空格或大写字母。

```yaml


# 错误
name: My Cool Skill



# 正确
name: my-cool-skill
```

### 技能未触发

症状：技能从不自动加载。

解决方法：修改您的描述字段。请参阅《描述字段》部分了解好坏示例。

快速检查清单：

- 是否过于通用？"帮助处理项目"不会起作用。
- 是否包含用户实际会说触发短语？
- 如果适用，是否提及相关文件类型？

调试方法：

询问Claude："您会在什么时候使用[技能名称]技能？"Claude会引用描述内容。根据缺失的内容进行调整。

### 技能触发过于频繁

症状：技能与无关查询一起加载。

解决方案：

1. 添加负面触发器。

```yaml
description: Advanced data analysis for CSV files. Use for statistical modeling, regression, clustering. Do NOT use for simple data exploration (use data-viz skill instead).
```

2. 更加具体。

```yaml


# 过于宽泛
description: Processes documents



# 更具体
description: Processes PDF legal documents for contract review
```

3. 明确范围。

```yaml
description: PayFlow payment processing for e-commerce. Use specifically for online payment workflows, not for general financial queries.
```

### MCP连接问题

症状：技能加载但MCP调用失败。

检查清单：

1. 验证MCP服务器是否已连接。
   - Claude.ai：设置 > 扩展 > [您的服务]。
   - 应显示"已连接"状态。
2. 检查身份验证。
   - API密钥有效且未过期。
   - 已授予适当的权限或范围。
   - OAuth令牌已刷新。
3. 独立测试MCP。
   - 要求Claude直接调用MCP而不使用技能。
   - 示例："使用[服务]MCP获取我的项目。"
   - 如果失败，问题是MCP，不是技能。
4. 验证工具名称。
   - 技能引用正确的MCP工具名称。
   - 检查MCP服务器文档。
   - 工具名称区分大小写。

### 未遵循指令

症状：技能加载但Claude不遵循指令。

常见原因：

1. 指令过于冗长。
   - 保持指令简洁。
   - 使用项目符号和编号列表。
   - 将详细参考移至单独的文件。
2. 指令被埋没。
   - 将关键指令放在顶部。
   - 使用`## 重要`或`## 关键`标题。
   - 如有必要，重复关键点。
3. 模糊的语言。

不好：

```text
Make sure to validate things properly.
```

好：

```text
关键：在调用create_project之前，验证：

- 项目名称非空
- 至少分配一个团队成员
- 开始日期不是过去的日期
```

高级技术：对于关键验证，考虑捆绑一个以编程方式执行检查的脚本，而不是依赖语言指令。代码是确定性的；语言解释不是。请参阅Office技能了解此模式的示例。

4. 模型"懒惰"：添加明确的鼓励。

```markdown


# 性能说明

- 花时间彻底完成此操作。
- 质量比速度更重要。
- 不要跳过验证步骤。
```

注意：将其添加到用户提示中比添加到`SKILL.md`更有效。

### 大上下文问题

症状：技能似乎变慢或响应质量下降。

原因：

- 技能内容过大
- 同时启用的技能过多
- 加载所有内容而非渐进式披露

解决方案：

1. 优化`SKILL.md`大小。
   - 将详细文档移至`references/`。
   - 链接到参考而不是内联。
   - 保持`SKILL.md`在5,000字以下。
2. 减少启用的技能。
   - 评估是否同时启用了超过20-50个技能。
   - 建议选择性启用。
   - 考虑为相关功能使用技能包。

## 安全注意事项

允许：

- 任何标准YAML类型，包括字符串、数字、布尔值、列表和对象
- 自定义元数据字段
- 长描述，最多1024个字符

禁止：

- XML尖括号（`<`或`>`），由于安全限制
- YAML中的代码执行；使用安全的YAML解析
- 以"claude"或"anthropic"为前缀命名的技能；这些是保留的
