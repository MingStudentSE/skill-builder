# Skill 构建器标准

本文件是 `skill-builder` 的执行标准。创建、改进或打包 skill 前，用它作为质量门。

完整来源资料位于 `references/source-books/`。本文件保留可直接执行的标准；完整书稿用于查证依据、寻找例子、更新规则。

## 核心定义

Agent Skill 是一个可移植文件夹，用来教会 agent 某个可重复工作流、领域方法、工具使用方式或决策流程。

必需结构：

```text
skill-name/
  SKILL.md
```

可选资源：

```text
skill-name/
  scripts/
  references/
  assets/
```

`SKILL.md` 必须包含带 `name` 和 `description` 的 YAML frontmatter，然后是 Markdown 指令正文。

## 渐进式披露

按三层设计每个 skill：

1. 元数据：`name` 和 `description` 总是可见，用于判断是否加载 skill。
2. 正文：只有当 skill 相关时，才加载 `SKILL.md`。
3. 配套资源：只有需要时才读取链接文件。

保持正文轻量。长政策、例子、API 说明、风格指南、schema 和 rubric 应放入 `references/`。

### 500-2000 字正文质量门

`SKILL.md` 正文的推荐区间是 500-2000 字。这个区间通常足够写清核心流程、关键标准和少量示例，又不会让关键信息被大量细节淹没。

太短的 skill 容易只剩泛泛建议；太长的 skill 会稀释关键信息权重，占用上下文，并让 agent 难以抓住执行重点。不要把 AI 已经具备的基础能力写进 skill；skill 应像菜谱一样给出本任务的关键火候、比例、顺序和偏好，而不是教人怎么握刀、怎么开火。

长度处理规则：

- 低于 500 字：检查是否缺少关键流程、标准、边界或示例；若任务本身极简单，可以保留。
- 500-2000 字：推荐区间。
- 超过 2000 字：触发压缩/下沉审查，优先删除基础常识、合并重复规则、把长例子和背景材料移到 `references/`。
- 超过 3000 字：触发拆分审查，优先按可独立执行的任务阶段拆成多个 skill，或改成主路由 skill + 子 skill。

审查顺序：

1. 判断是否混入多个独立任务阶段，例如“选题 + 调研 + 初稿 + 审校 + 配图”。
2. 如果用户可能只需要其中一个阶段，优先拆成多个 skill，或改成主路由 skill + 子 skill。
3. 如果内容是长例子、rubric、背景资料、模板说明或 API 说明，下沉到 `references/`。
4. 如果内容是重复且确定性的操作，移到 `scripts/`。
5. 只有每次执行都必须读取的核心流程、决策规则、关键标准、用户偏好和安全边界，才保留在 `SKILL.md`。

超过 2000 字不是自动失败，但必须说明已压缩/已下沉，或为什么需要保留。超过 3000 字不是自动失败，但必须说明已拆分，或为什么暂时保留为单一 skill。

## Frontmatter 标准

`name` 字段：

- 使用 kebab-case。
- 与文件夹名称一致。
- 避免大写、空格、下划线和平台保留词。

`description` 字段：

- 说明 skill 做什么。
- 说明什么时候使用。
- 包含具体触发短语或任务类别。
- 当文件类型、工具、平台或数据形态会影响触发时，明确写出。
- 少于 1024 字符。
- 避免 XML 尖括号。
- 使用第三人称描述，例如“用户想要……时，应使用本 skill”。

好的 description 同时包含能力、触发和边界：

```yaml
description: 用于审查和改进 Agent Skills 的触发精度、渐进式披露、资源组织和打包准备度。用户要求审查 SKILL.md、优化 skill 描述、重构资源或准备分发 skill 时，应使用本 skill。
```

弱 description 过于模糊：

```yaml
description: 帮助处理 skills。
```

## 正文标准

正文要写给之后执行任务的 agent。

使用：

- 清晰的命令式步骤。
- 决策规则。
- 触发正例和反例。
- 期望输出。
- 安全边界。
- 指向配套 references、scripts 或 assets 的链接。

避免：

- 营销文案。
- 长背景文章。
- 在正文中重复详细引用材料。
- 基础模型本来就知道的泛泛建议。
- 对工具或运行环境做隐含假设。

## 资源放置规则

按下表放置内容：

| 内容类型 | 放置位置 | 原因 |
| --- | --- | --- |
| 触发、工作流、决策树 | `SKILL.md` | 每次 skill 运行都需要 |
| 长例子、rubric、风格指南 | `references/` | 只在需要时加载 |
| API 文档、schema、政策 | `references/` | 内容大，属于查阅材料 |
| 确定性重复代码 | `scripts/` | 更可靠，也更省 token |
| 带重试的外部 API 调用 | `scripts/` | 错误处理更稳定 |
| 模板、样板代码、字体、图片 | `assets/` | 用于输出，不是指令 |

删除无用的脚手架示例。

## 六种设计模式

可单独使用，也可组合。选择能解决重复工作流的最简单模式；好的 skill 会让 agent 的下一步行动非常明确。

| 模式 | 适用场景 | 典型流程 | 核心质量门 |
| --- | --- | --- | --- |
| 检查清单型 Checklist | 审校、代码 review、合规检查、质量验证 | 检查 -> 标记 -> 修正 -> 验证 | 条目具体到可执行，结果必须带动作 |
| 多方案选择型 Options | 选题、命名、设计风格、策略选择 | 生成 3-4 个方案 -> 对比 -> 用户选择 -> 执行 | 方案之间有真实差异，优劣分析诚实 |
| 多阶段流水线型 Pipeline | 端到端大任务、长文档、复杂制作流程 | 阶段输入 -> 文件产出 -> 检查点 -> 下一阶段 | 每阶段有文件产出，可从断点恢复 |
| 外部 API 集成型 Integration | API、MCP、CLI、上传、发消息、文档写入 | 准备 -> 小步调用 -> 错误处理 -> 反馈 | 密钥走环境变量，大操作拆小步骤 |
| 多 Agent 协作型 Swarm | 可真正并行的大规模任务 | 依赖分析 -> 分配 -> 并行产出 -> 人工 review 合并 | 子任务必须独立，用文件/git 协调 |
| 思维蒸馏型 Distillation | 提取人物、团队或领域方法，形成思维引擎 | 收集证据 -> 提取框架 -> 验证 -> 编码为 skill | 蒸馏思维框架，不做浅层语气模仿 |

### 模式 1：检查清单型 Checklist

用于有明确质量门的任务。清单条目必须具体，不写“检查语言是否自然”这类模糊指令；要写可判断、可删除、可替换的标准。

设计要点：

- 条目具体到可执行，例如给出要删除的词汇、要标记的句式、要验证的字段。
- 分优先级，先查事实错误、合规风险、结构性缺陷，再查风格和润色。
- 分轮次检查，每轮聚焦少数维度，避免一次性检查导致顾此失彼。
- 检查结果必须带动作：标记问题后给出修改建议或直接改写版本。

### 模式 2：多方案选择型 Options

用于需要用户在多个方向中做选择的任务。AI 负责生成和分析，决策权默认留给用户。

设计要点：

- 方案数量优先 3-4 个；2 个太少，5 个以上容易制造决策疲劳。
- 方案之间要有真实差异，不只是同一想法的不同措辞。
- 每个方案必须有至少一条明确劣势，避免把所有方案都夸成“很好”。
- 加入现实约束评估：工作量、时间、资源、风险、差异化空间。

### 模式 3：多阶段流水线型 Pipeline

用于端到端工作流。把大任务拆成多个阶段，每阶段有明确输入、输出和检查点。

设计要点：

- 每个阶段必须有文件产出，不让关键状态只存在于对话上下文。
- 阶段之间设置检查点，完成后汇报并等待确认，减少方向偏差的返工成本。
- 支持从任意阶段恢复：开头检测项目文件、阶段产物和当前进度。
- 给出进度可视化，例如“已完成 3/5 阶段，当前：第 7 章写作”。

### 模式 4：外部 API 集成型 Integration

用于调用外部服务。难点通常不是业务逻辑，而是认证、调用顺序、重试和边界情况。

设计要点：

- 密钥永远从环境变量或安全配置读取，不在 skill 中硬编码。
- 写清需要哪些环境变量，以及用户在哪里获取。
- 大操作拆成小步骤，每步有成功/失败反馈，可从失败步骤重试。
- 记录已知 API 坑点、调用顺序、限制条件和代理/网络注意事项。
- 每个黑盒调用都向用户反馈进度和结果。

### 模式 5：多 Agent 协作型 Swarm

仅当任务规模足够大、子任务真正独立时使用。并行有启动和合并成本，不是所有任务都值得 swarm。

设计要点：

- 先画依赖关系；有依赖箭头的任务不要强行并行。
- 用文件系统、git worktree、branch 或共享产物协调，不让 agent 之间复杂对话。
- 每个 agent 的任务边界、输入、输出文件和禁止改动范围要明确。
- 合并阶段需要人工 review；不要简单拼接并行产物。
- 小任务不要并行，避免拆分和合并成本超过收益。

### 模式 6：思维蒸馏型 Distillation

用于把人物、团队或领域的方法论蒸馏成可运行的思维框架。目标不是模仿语气，而是复现判断框架、关注变量和决策启发式。

设计要点：

- 先收集足够证据，再提炼心智模型、决策规则、反例和表达 DNA。
- 区分“语气模仿”和“思维框架运行”；不要只做角色扮演。
- 保留来源地图和适用边界，避免把人物观点绝对化。
- 用真实问题测试蒸馏结果：能否给出稳定、可解释、可反驳的判断。
- 辅助视角只能提供判断材料，不应替用户接管目标或结构。

## 创建循环

遵循这个循环：

1. 澄清具体例子。
2. 规划可复用内容。
3. 检查 Anthropic 官方 `skill-creator` 是否已安装；如果没有，先从官方仓库安装。
4. 用官方 `skill-creator` 初始化。
5. 编辑 `SKILL.md` 和配套资源。
6. 校验结构和触发质量。
7. 用正例和反例测试。
8. 准备好后打包。
9. 根据真实使用反馈迭代。

官方 `skill-creator` 来源：

```text
https://github.com/anthropics/skills/tree/main/skills/skill-creator
```

## 官方 skill-creator 协作协议

`skill-builder` 和官方 `skill-creator` 的分工：

- `skill-builder` 负责写什么、怎么拆、采用哪种设计模式、哪些内容下沉、如何定义质量门。
- 官方 `skill-creator` 负责结构合法性、脚本校验、打包、必要时的触发评测。

固定协作流程：

1. `skill-builder` 理解需求或现有 skill 的问题。
2. `skill-builder` 判断设计模式和是否需要拆分。
3. `skill-builder` 形成构建/优化简报。
4. `skill-builder` 建立官方 creator 脚本能力矩阵。
5. 官方 creator 初始化新 skill；若无初始化脚本，手动创建最小结构。
6. `skill-builder` 编写或改进 `SKILL.md`、`references/`、`scripts/`、`assets/`。
7. 官方 creator 执行 `quick_validate.py`。
8. `skill-builder` 根据错误和质量门修复。
9. 需要分发时，官方 creator 执行 `package_skill.py`。
10. `skill-builder` 汇报设计决策、校验结果和包路径。

## 优化已有 skill 流程

优化已有 skill 时，不要默认重写。先诊断，再最小修改。

诊断顺序：

1. 触发：`description` 是否能命中正例、避开反例。
2. 长度：正文是否在 500-2000 字；是否有基础常识、长例子、背景材料淹没重点。
3. 模式：是否能归入六种模式；是否把多个独立模式硬塞进一个 skill。
4. 结构：长材料是否下沉到 `references/`，确定性操作是否进入 `scripts/`。
5. 安全：是否有硬编码密钥、本机绝对路径、未确认的删除/发布/外部 API 操作。
6. 可恢复：Pipeline / Integration / Swarm 是否有文件产出、状态判断和失败重试。
7. 校验：官方 `quick_validate.py` 是否通过。

常见优化动作：

| 问题 | 优先动作 |
| --- | --- |
| 触发太泛 | 收紧 `description`，加入反向边界 |
| 不该触发 | 增加不适用场景，改窄任务类别 |
| 正文过长 | 删除基础常识，下沉长例子/rubric，必要时拆 skill |
| 流程太抽象 | 加入具体输入、输出、判断标准和失败处理 |
| 资源散乱 | 只保留被 `SKILL.md` 引用且确实有用的资源 |
| 集成脆弱 | 把 API 调用、重试和错误处理脚本化 |
| 多阶段易中断 | 增加阶段产物、状态检测和断点恢复 |

优化完成后必须说明：改了什么、为什么这样改、是否影响触发范围、校验是否通过。

## 触发基准测试

准备两组例子。

正向触发：

- 直接请求：“创建一个 skill”“把这个工作流做成 skill”。
- 间接请求：“我总是重复做这件事，能不能做成可复用能力？”
- 维护请求：“修这个 SKILL.md”“打包这个 skill”。

反向触发：

- 没有要求可复用 skill 行为的一般 coding 任务。
- 只是一次性 prompt，不需要沉淀工作流。
- 只要求安装已完成 skill，且不需要编辑、校验或打包。

持续优化 `description`，直到它能抓住正例，同时不误触发反例。

## 安全和确认

这些操作前必须明确确认：

- 发布或上传到外部平台。
- 发送消息或执行对外可见 API 调用。
- 删除用户创建的 skill 内容。
- 覆盖一个归属不明确的已有 skill。

不要硬编码 secret。token 和凭据必须使用环境变量或用户当前环境的安全配置。

不要在 skill 文件中写入用户机器绝对路径。需要引用工具位置时，使用相对路径、环境变量、工具发现结果，或让当前执行 agent 在运行时定位。

## 打包前检查清单

- `SKILL.md` 存在，且 YAML frontmatter 合法。
- 文件夹名称等于 `name`。
- `description` 具体且触发明确。
- 正文保持可控，长材料放入引用文件。
- `SKILL.md` 正文优先保持在 500-2000 字；超过 2000 字已完成压缩/下沉审查，超过 3000 字已完成拆分审查，并记录处理结果。
- 没有无意保留的示例文件。
- 所有被引用的文件都存在。
- 如果有脚本，要么可执行，要么明确写出解释器调用方式。
- 校验通过。
- zip 包由官方打包流程生成。

## 常见失败模式

| 失败情况 | 修正方式 |
| --- | --- |
| skill 不触发 | 用更具体的触发短语重写 `description` |
| skill 触发过多 | 增加边界和反向任务类别 |
| 输出太泛 | 增加领域步骤、例子和质量门 |
| `SKILL.md` 过长 | 推荐 500-2000 字；超过 2000 字先压缩/下沉，超过 3000 字触发拆分审查；按独立任务阶段拆成多个 skill，或把详细材料移到 `references/` |
| API 工作流脆弱 | 把调用写入带重试和错误处理的脚本 |
| 打包失败 | 严格按官方校验输出修复 |
